# インスタンス・メソッドの作りかた

## キーポイント

* 属性キーワード`static`の付いたメンバーを「スタティック・メンバー」という
* 「スタティック・フィールド」は **クラスの共有メンバー** になる
* 「スタティック・メソッド」は **staticの付いたメンバーだけ** を読み書きできる
* 属性キーワード`static`のないメンバーを「インスタンス・メンバー」という
* 「インスタンス・フィールド」は **クラス変数ごとに固有のメンバー** になる
* 「インスタンス・メソッド」は **全てのメンバー** を読み書きできる
* 「インスタンス・メソッド」は、フィールドを読み書きするプログラムを **かたまり** にするもの

## 1 スタティックメンバーとインスタンスメンバー

### 1.1 static(スタティック)キーワードの効果

メンバーの前に付く`static`(スタティック)は「属性キーワード」の一種です。<br>
`static`の付いたメンバーは **クラスの共有メンバー** となり、その型の変数を作らなくても使えます。<br>
`static`のないメンバーは **クラス変数ごとのメンバー** となり、変数を作らなくては使えません。

`static`の付いたフィールドを「スタティック・フィールド」、ないフィールドを「インスタンス・フィールド」といいます。

> **【インスタンス】**<br>
> `new`で作る変数のことを「インスタンス、「実例」「具体例」という意味)といいます。

例えば、以下のようなクラス`Test`があるとします。

```c#
public class Test
{
  public int Number; // インスタンス・フィールド(共有されない)
  public static int StaticNumber; // スタティック・フィールド(共有される)
}
```

<div style="page-break-after: always"></div>

このとき、`static`の付いた`StaticNumber`フィールドは、次のように読み書きできます。

```c#
Test.StaticNumber = 57;
Console.WriteLine(Test.StaticNumber); // 57が出力される
```

これに対して、`static`のない`Number`フィールドは、次のように変数を宣言する必要があります。

```c#
// Test.Number = 57; // NG. Number はクラスの共有メンバーではない

Test t = new(); // Test型の変数 t を宣言
t.Number = 57;  // OK. 変数 t のメンバーに代入
Console.WriteLine(t.Number); // OK. 変数 t のメンバーを出力
```

インスタンス・フィールドは、変数ごとに新しく作成されます。<br>
そのため、フィールド名が同じでも、変数が違えば別のフィールド変数として扱われます。

```c#
Test s = new();
Test t = new();
s.Number = 13;  // 変数 s の Number フィールドに 13 を代入
t.Number = 597; // 変数 t の Number フィールドに 597 を代入
Console.WriteLine(s.Number); // 13 が出力される
Console.WriteLine(t.Number); // 597 が出力される
```

`static`の有無は次のように使い分けます。

* クラス全体でひとつだけ必要: `static`を付ける
* 配列にするなど、複数の変数を作りたい: `static`を付けない

「クラス全体でひとつだけ必要」なことは少ないため、大抵のフィールドには`static`を付けません。<br>
`static`を付けるのは、 **アプリ全体でひとつだけ必要なクラス** だけです。<br>
この例としては、Visual Studioが自動的に作成する`Program`クラスが挙げられます。

<div style="page-break-after: always"></div>

### 1.2 staticの有無とメソッド

`static`の有無は、フィールドだけでなくメソッドでも同様に働きます。

* `static`の付いたメソッド: `static`なメンバーだけ使える
* `static`のないメソッド: `static`の有無に関わらず全てのメンバーを使えるが、変数が必要

`static`の付いたメソッドを「スタティック・メソッド」、ないメソッドを「インスタンス・メソッド」といいます。

次のプログラムは、インスタンスメソッドとスタティックメソッドの **読み書きできるフィールドの違い** を示しています。

```c#
public class Test
{
  public int Number; // staticじゃないフィールド
  public static int StaticNumber; // staticなフィールド

  // インスタンス・メソッド
  public void WriteNumbers()
  {
    Console.WriteLine(Number); // OK. static じゃないフィールドを読み書きできる
    Console.WriteLine(StaticNumber); // OK. static なフィールドも読み書きできる
  }

  // スタティック・メソッド
  public static void WriteStatic()
  {
    //Console.WriteLine(Number); // NG. static じゃないフィールドは読み書きできない
    Console.WriteLine(StaticNumber); // OK. static フィールドなら読み書きできる
  }
}
```

フィールドと同様に、ほとんどのメソッドには`static`を付けません。<br>
一般的には、メソッドに`static`を付けるのは、 **アプリ全体でひとつだけ必要なクラスだけ** です。<br>
この例としては、Visual Studioが自動的に作成する`Program`クラスが挙げられます。

>`Main`メソッドには必ず`static`が付きます。

### 1.3 インスタンス・メソッドの使いみち

メソッドを作る目的は以下の2つです。

1. プログラムのかたまりに名前を付けて覚えやすくする
2. 同じプログラムを何度も書かなくて済むようにする(再利用を可能にする)

メソッドを作るとき、スタティックメンバーを使うプログラムはスタティックメソッドとして作ります。<br>
インスタンスメンバーをを使うプログラムならインスタンスメソッドとして作ります。

例えば、次のプログラムでは3つの座標変数を持つ`Position`クラスを定義しています。<br>
`Position`クラスには`Add`(アド)と`Write`(ライト)という2つのメソッドがあります。<br>
これらのメソッドはインスタンスフィールドを操作するので、インスタンスメソッドとしています。

```c#
public class Position
{
  public float X;
  public float Y;
  public float Z;

  // コンストラクタ
  public Position(float x, float y, float z)
  {
    X = x;
    Y = y;
    Z = z;
  }

  // 全てのフィールドに f を足すメソッド
  public float Add(float f)
  {
    X += f;
    Y += f;
    Z += f;
  }

  // フィールドの内容を出力するメソッド
  public void Write()
  {
    Console.WriteLine("(" + X + "," + Y + "," + Z + ")");
  }
}
```

<div style="page-break-after: always"></div>

この`Position`クラスは、次のように使うことができます。

```c#
// メソッドを使ったプログラム
Position a = new(1, 2, 3);
a.Add(5);
a.Write();
a.Add(-3);
a.Write();
a.Add(8);
a.Write();
```

もしも、インスタンスメソッドを使わずに同じ結果を得るには、次のように書くことになります。

```c#
// 上と同じ内容を、メソッドを使わずに書くと⋯
Position a = new(1, 2, 3);
a.X += 5;
a.Y += 5;
a.Z += 5;
Console.WriteLine("(" + a.X + "," + a.Y + "," + a.Z + ")");
a.X += -3;
a.Y += -3;
a.Z += -3;
Console.WriteLine("(" + a.X + "," + a.Y + "," + a.Z + ")");
a.X += 8;
a.Y += 8;
a.Z += 8;
Console.WriteLine("(" + a.X + "," + a.Y + "," + a.Z + ")");
```

メソッドを使わないプログラムは、行ごとに見れば「何をしているか」は分かりやすいです。<br>
ですが、全体としては読みにくく、メソッドを使った場合と比べてプログラムが長くなります。<br>
そのため、「メソッドがやること」を知っていれば、メソッドを使ったプログラムのほうが理解しやすいです。

それから、足す数や出力方法を変えたい場合、メソッドを使わないプログラムでは書き直しが大変です。<br>
メソッドを使ったプログラムでは、修正や変更はメソッドを書き直すだけです。行ごとに書き直す必要はありません。

このように、 **同じことをするプログラムをメソッドにまとめる** という方法は、メソッドの基本的な使いかたです。

### 2.1 頭がブロックにぶつかるようにする

ブロックの下からジャンプすると、ブロックをすり抜けてブロックの上に着地できてしまいます。そういうゲーム性も悪くありませんが、今回の`PenguinsJump`アプリでは、ちゃんと頭をぶつけられるようにします。

頭の判定はジャンプの上昇中、つまり`JumpSpeed`が`0`より大きい場合だけ行います。

```diff

       float blockR = blockL + blockList[a].Size * 64.0f; // ブロックの右側の X 座標
       float blockT = blockList[a].Y * 64.0f;             // ブロックの上側の Y 座標
       float blockB = blockT + 64.0f;                     // ブロックの下側の Y 座標
+
+      // 上昇中だけ頭を判定する
+      if (player.JumpSpeed > 0.0f)
+      {
+        // プレイヤーの頭の座標を計算
+        float topX = player.X + 32.0f; // 頭のX座標
+        float topY = player.Y + 8.0f;  // 頭のY座標
+
+        // 頭がブロックと重なっていたら、頭をぶつけたことにする
+        if (topX >= blockL && topX < blockR &&
+            topY >= blockT && topY < blockB)
+        {
+          player.Y = blockB - 8.0f; // プレイヤーをブロックの下に強制移動
+          player.JumpSpeed = 0;     // ジャンプ速度を0にする
+        }
+      }

       // 上昇中でなければ着地を判定する
       if (player.JumpSpeed <= 0.0f)
```

頭の判定は上昇中のみ実行します。<br>
下降中に頭がブロックと重なったとき、急に落下速度が0になるバグを起こしたくはないでしょう。

頭の判定は、中央上部の1点だけです。足元のように横長にしない理由はあとで説明します。<br>
また、Y座標は「画像の上端から`8`ドット下」にずらしています。<br>
こうすると、ぶつかったときの納得感が高くなります。

### 2.2 プレイヤーの左側がブロックに当たるようにする

頭と同様に、体の左右にもブロックとの判定を付けます。<br>
ですが、その前に、左右の判定が必要な理由を確認しましょう。ブロックリストに、データを1つを追加してください。

```diff
     private static Block[] blockList = {

       new(6.0f, 5.0f, 3), new(10.0f, 3.0f, 8),
+      new(1.0f, 9.0f, 1),

```

プログラムが書けたら`>PenguinsJump`ボタンをクリックしてアプリを実行してください。<br>
スタート地点の左にブロックが増えているので、プレイヤーを横からぶつけてください。<br>
すると、何も無いかのようにブロックをすり抜けてしまいます。

また、空中ブロックにななめ下からジャンプしてぶつかると、不自然な位置で頭をぶつけたりします。<br>
これらの現象は、左右の判定がないために起きています。左右の判定を付けると、これらの現象を防げます。

それでは、判定を付けましょう。着地を判定するプログラムの下に、次のプログラムを追加してください。

```diff

+

+      float rightX = player.X + 56.0f; // 右側のX座標

+      float sideYB = player.Y + 52.0f; // 側面下側のY座標

+          sideYB >= blockT && sideYT < blockB)
+      {
+        player.X = blockR - 8.0f; // プレイヤーをブロックの右に強制移動
+      }
+
+      // 右側の衝突判定
+      if (rightX >= blockL && rightX < blockR &&
+          sideYB >= blockT && sideYT < blockB)
+      {
+        player.X = blockL - 56.0f; // プレイヤーをブロックの左に強制移動
+      }
     } // プレイヤーとブロックの衝突の終わり
   } // UpdatePlayメソッドブロックの終わり
```

側面の判定には速度制限を付けません。<br>
プレイヤークラスには横方向の速度パラメータがないので、制限したくてもできないからです。

側面の判定は、画像の端より8ドット内側にしています。<br>
また、上下方向も画像の高さいっぱいではなく、12ドットずつ内側にしてあります。<br>
画像の端にはあまり絵が描かれていないため、少し狭めの範囲にするほうが視覚的に納得しやすくなります。

それから、「頭が上のブロックにぎりぎりぶつからないように」ジャンプしてみてください。<br>
すると、プレイヤーはブロックを避けるように、するりと横に移動します。<br>

### 2.3 衝突判定をクラスにする

ここまで、足元、頭、左側、右側の4つの衝突判定を作りました。<br>
この4つをよく見ると、if文の部分はどれも似ています。こういうのはメソッドにするのがよいです。

また、構造的には「ブロックの範囲と、別の何かの範囲の交差」という判定になっています。<br>
こういう場合は、「範囲」をクラスにして、衝突判定は「範囲」のメソッドにすると扱いやすいです。

「範囲」は上下左右の4つの座標を持つ長方形として定義できます。<br>
そこで、クラス名は`Box`(ボックス)としましょう。<br>

この`Box`クラスに、衝突判定を行うメソッドを追加します。<br>

メソッドの結果は「衝突した・しない」の二択なので、戻り型は`bool`がいいでしょう。<br>

### 2.4 ボックスと点の衝突判定メソッドを追加する

頭の判定は1点だけなので、ボックス同士の衝突判定はやりすぎです。<br>
そこで、`Box`と点の衝突判定を行うメソッドも追加しましょう。<br>
メソッド名と戻り型は、`Box`同士の衝突判定メソッドと同じでいいでしょう。

### 2.11 トゲブロックの衝突判定

古来から、2Dゲームでは「トゲに触れたらダメージを受ける」と決まっています。<br>
ですが、現在はプレイヤーがトゲに触れても何も起きません。まだトゲの衝突判定がないからです。

そこで、トゲの衝突判定を書いて、プレイヤーに死を与えましょう。<br>
トゲの衝突範囲は、画像の右から4ドット、左から8ドット離します。上下は`16`～`64`ドットとします。

<div align="center"><img alt="" src="images/11_spike_up_hitbox.png" /><br>[トゲの衝突判定の範囲]</div>

衝突判定に使うプレイヤーの範囲は、ゴール判定で作った`Box`が流用できます。<br>
プレイヤーの衝突範囲を計算するプログラムの下に、次のプログラムを追加してください。

```diff

     Box boxPlayer = new(
       player.X + 12.0f, player.X + 52.0f,
       player.Y + 8.0f, player.Y + 56.0f);
+
+    // プレイヤーとトゲブロックとの接触
+    for (int a = 0; a < spikeList.Length; a += 1)
+    {
+      トゲブロックの上下左右の座標を計算
+      Box boxSpike = new(
+        spikeList[a].X * 64.0f +  4.0f, (spikeList[a].X + spikeList[a].Size) * 64.0f - 8.0f,
+        spikeList[a].Y * 64.0f + 16.0f, spikeList[a].Y * 64.0f + 64.0f);
+
+      // プレイヤーとトゲブロックが重なっていたら死亡状態にする
+      if (boxSpike.Intersect(boxPlayer))
+      {

+      }
+    } // プレイヤーとトゲブロックとの接触の終わり

     // ゴールの衝突範囲を計算

       goal.X, goal.X + 64.0f,
       goal.Y, goal.Y + 64.0f);
```
